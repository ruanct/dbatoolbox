# Oracle 19c 与 Oracle 11g 全面对比

> **说明**：Oracle 11g（尤其是 11.2.0.4）已于 2021 年 12 月 31 日结束 Premier Support，而 19c 是 Oracle 的长期支持版本（LTS），支持期至少到 2027 年 4 月。以下从多个维度进行详细对比。

## 一、架构层面

### 1.1 数据库架构

| 对比项               | Oracle 11g              | Oracle 19c                               |
| :------------------- | :---------------------- | :--------------------------------------- |
| **核心架构**         | 传统单实例/单数据库架构 | 引入**多租户架构**（CDB + PDB）          |
| **实例与数据库关系** | 一对一或多对一（RAC）   | CDB 可承载多个 PDB，实现一对多           |
| **非CDB模式**        | 默认即为非CDB           | 仍支持非CDB模式，但官方推荐使用CDB       |
| **资源隔离**         | 实例级隔离              | PDB 之间相互隔离，各自拥有独立的数据文件 |

> 多租户是 19c（自 12c 起）最显著的架构变革，CDB 作为容器数据库，可以承载多个可插拔数据库（PDB），每个 PDB 在应用层面可视为一个独立的数据库。

### 1.2 存储引擎

| 对比项           | Oracle 11g           | Oracle 19c                                                   |
| :--------------- | :------------------- | :----------------------------------------------------------- |
| **存储结构**     | 基于分区表 + B树索引 | 多模型数据库存储引擎（ODMDS），支持文档型、关系型等多种存储结构 |
| **表空间管理**   | 传统表空间管理       | ASM（自动存储管理）更成熟                                    |
| **VARCHAR2上限** | 最大 4000 字节       | 可支持最大 **32767 字节**                                    |

## 二、性能与优化器

### 2.1 优化器（核心差异）

| 对比项           | Oracle 11g              | Oracle 19c                                                   |
| :--------------- | :---------------------- | :----------------------------------------------------------- |
| **优化器技术**   | 基于成本的优化器（CBO） | 引入**自动化机器学习技术的 SQL 优化器**                      |
| **执行计划**     | 静态执行计划            | **自适应执行计划**（Adaptive Execution Plan），运行时动态调整 |
| **自动索引**     | 无                      | **自动索引创建**功能（可自动提议、创建、验证和删除索引）     |
| **实时统计信息** | 手动或定时收集          | **实时统计信息收集**（部分功能依赖一体机）                   |
| **SQL隔离**      | 无                      | **SQL隔离**功能（部分功能依赖一体机）                        |

> **升级注意事项**：从 11g 升级到 19c 后，大多数查询性能会提升，但少数关键查询可能因优化器行为变化而出现性能衰退，严重时可能慢 **1000 倍**。主要原因是统计信息未重收集、直方图缺失或采样失真——这三类问题占性能衰退案例的 **80% 以上**。

### 2.2 整体性能

- 19c 在查询优化、内存管理、自动化运维方面全面优于 11g
- 19c 优化了内存管理和自动化索引管理
- 部分场景（如特定客户端的 fetch 操作）19c 可能比 11g 慢约 50%，需针对性调优

## 三、安全特性

| 对比项            | Oracle 11g     | Oracle 19c                                                   |
| :---------------- | :------------- | :----------------------------------------------------------- |
| **加密算法**      | 较旧的加密标准 | **AES-256 默认**、TLS 1.3 支持                               |
| **数据脱敏**      | 功能有限       | **Transparent Data Encryption with Data Redaction**（透明数据加密 + 数据修订） |
| **密码策略**      | 基础密码验证   | 更严格的密码验证策略和审计功能                               |
| **表空间加密**    | 不支持         | 支持多租户加密隔离                                           |
| **行级/列级安全** | 部分支持       | 显示/隐藏列、行级安全性增强                                  |
| **红蓝分离**      | 无             | 引入**数据红蓝分离机制**，将生产与测试系统网络隔离           |
| **密钥管理**      | 分散管理       | 统一密钥管理                                                 |

> **安全评分变化**：根据第三方安全评估，从 11g 到 18c 安全性明显提升，19c 有所回落至接近 11g 水平，21c 和 23c 再次提升。

## 四、高可用性与 Data Guard

| 对比项               | Oracle 11g            | Oracle 19c                                              |
| :------------------- | :-------------------- | :------------------------------------------------------ |
| **备库DML操作**      | 备库只读，不能执行DML | **Active Data Guard DML自动重定向**，备库可直接执行事务 |
| **并行重做日志应用** | 单实例应用            | **多实例并行重做日志应用**，提升同步效率                |
| **备库智能性**       | 被动接收日志          | 更智能的备库管理                                        |

## 五、开发与数据类型

### 5.1 JSON 支持

| 对比项       | Oracle 11g                | Oracle 19c                                     |
| :----------- | :------------------------ | :--------------------------------------------- |
| **JSON支持** | 几乎不支持（12c开始引入） | 原生JSON支持，包括JSON对象映射、JSON合并更新等 |
| **JSON存储** | 需以文本存储              | 支持JSON存储对象，可将JSON数据作为对象存储     |

### 5.2 SQL 与 PL/SQL

| 对比项            | Oracle 11g     | Oracle 19c                                     |
| :---------------- | :------------- | :--------------------------------------------- |
| **LISTAGG函数**   | 基础功能       | 支持 **DISTINCT** 关键字去重                   |
| **wm_concat函数** | 支持（非官方） | **已移除**，需改用 LISTAGG 或自定义函数        |
| **区块链表**      | 无             | 支持创建不可变表（Immutable Tables）和区块链表 |
| **并行CTAS**      | 有限支持       | 支持并行执行 CREATE TABLE AS SELECT            |
| **REST支持**      | 无             | REST Enabled SQL 支持                          |

### 5.3 分区表

| 对比项           | Oracle 11g     | Oracle 19c                                       |
| :--------------- | :------------- | :----------------------------------------------- |
| **分区操作**     | 单分区逐个维护 | 支持**多个分区同时维护**（如同时MERGE 12个分区） |
| **混合分区表**   | 不支持         | 支持**混合分区表**（Hybrid Partitioned Tables）  |
| **物化视图刷新** | 基础支持       | 分区表场景下 fast 刷新更稳健、限制更少           |

## 六、管理与运维

| 对比项                  | Oracle 11g          | Oracle 19c                                                   |
| :---------------------- | :------------------ | :----------------------------------------------------------- |
| **自动诊断**            | 自动诊断框架（ADF） | 诊断和修复功能进一步增强                                     |
| **SQL Plan Management** | 引入 SPM            | SPM 更成熟完善                                               |
| **数据泵**              | 数据泵增强          | 更多增强功能                                                 |
| **备份恢复**            | RMAN 基础功能       | 基于时间轴恢复、快速恢复等更丰富机制                         |
| **客户端连接**          | 支持旧版客户端协议  | 默认要求更高版本协议，旧客户端需配置 `SQLNET.ALLOWED_LOGON_VERSION` |

## 七、支持与生命周期

| 对比项               | Oracle 11g                  | Oracle 19c           |
| :------------------- | :-------------------------- | :------------------- |
| **Premier Support**  | **已于 2021年12月31日结束** | 有效支持中           |
| **Extended Support** | 已结束或即将结束            | 至少到 **2027年4月** |
| **安全补丁**         | 不再接收定期安全补丁        | 持续接收安全更新     |
| **合规性**           | 可能无法满足法规要求        | 符合最新合规标准     |
| **云就绪**           | 不支持云端部署              | 更容易迁移至云端     |

## 八、升级路径

从 11g 升级到 19c 的推荐路径：

1. **直接升级**：若当前为 **11.2.0.4**，可直接升级到 19c
2. **分步升级**：若版本更低，需先升级到 11.2.0.4、12.1.0.2 或 12.2.0.1
3. **升级工具**：可使用 Oracle 的 AutoUpgrade 工具进行升级和转换

## 九、总结

| 维度         | 11g 优势     | 19c 优势                                |
| :----------- | :----------- | :-------------------------------------- |
| **架构**     | 简单成熟     | 多租户灵活、资源利用率高                |
| **性能**     | 稳定可预期   | 自动化优化、自适应执行计划              |
| **安全**     | —            | 更强加密、数据脱敏、统一密钥管理        |
| **高可用**   | —            | 备库DML、并行日志应用                   |
| **开发**     | —            | JSON原生支持、更大VARCHAR2、更多SQL函数 |
| **运维**     | —            | 自动索引、实时统计、更丰富诊断          |
| **生命周期** | ❌ 已停止支持 | ✅ 长期支持至2027+                       |

> **建议**：如果现有 11g 系统运行稳定且无新业务需求，可暂缓升级；但考虑到安全补丁缺失、合规风险和技术债务累积，**建议尽早规划升级至 19c**。升级前务必在测试环境充分验证，重点关注 SQL 执行计划变化和应用程序兼容性。
