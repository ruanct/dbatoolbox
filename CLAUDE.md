# CLAUDE.md - Django 项目 AI 编码规范

本文档适用于当前 **Python Django** 项目，用于约束 Claude Code 所有代码编写、修改、重构、Bug 修复行为。统一项目编码风格、分层逻辑、模块职责，杜绝LLM常见的过度设计、层级混乱、代码乱写、改动泛滥等问题。

**作用**：用于约束 AI 代码生成、统一项目架构、编码风格、技术选型、规避错误方案，全程遵循现有项目结构迭代，禁止私自改版架构。

**取舍原则：优先严谨、规范、可维护、贴合Django官方设计思想，不盲目追求新奇写法；简单琐碎任务可灵活简化。**

## 一、通用 AI 编码强制准则

### 1. 编码前先思考，禁止主观臆断

*   明确列出实现前提与假设，存在业务、逻辑、需求不确定性必须主动提问，不私自敲定方案

*   需求存在多种解读、多种实现方案时，列出差异对比，不静默自选实现方式

*   发现更简单、贴合Django原生思想的优雅实现方式，主动说明并优先采用，拒绝过度设计

*   需求模糊、逻辑冲突、场景缺失、权限/边界不明确时，立即停止编码，明确疑点并等待确认

### 2. 极简优先，拒绝冗余 speculative 代码

*   只实现用户明确要求的功能，不额外扩展接口、不新增未要求的字段、不画蛇添足

*   单次使用、专属场景的逻辑，不做全局抽象封装、不预留无用扩展性

*   不私自增加未要求的配置项、兼容逻辑、工具方法、日志打印

*   不为业务上不可能出现的场景编写异常处理、容错逻辑

*   代码能精简则精简，杜绝堆砌式、冗余注水代码，避免复杂化Django原生简洁逻辑

### 3. 精准局部修改，最小改动原则

*   仅修改需求直接相关的代码、文件、逻辑，禁止主动优化、格式化、重构无关业务代码

*   原有正常运行、无Bug的Django路由、视图、模型、配置，禁止私自重构调整

*   严格匹配项目现有代码风格、命名、分层逻辑，不强行统一历史旧代码

*   发现无关死代码、冗余逻辑仅做文字提醒，不擅自删除、改动

*   仅清理本次改动产生的无用import、变量、函数、字段，原有冗余内容不动

*   所有代码改动必须可追溯用户需求，无关联的无效改动一律禁止

### 4. 目标驱动，可验证交付

*   所有任务转化为可验证目标，拒绝模糊交付、“能跑就行”

*   修复Bug：先编写复现用例，再修复代码，保证功能闭环

*   新增接口/功能：明确入参、出参、权限、校验、边界条件后再编码

*   重构代码：保证重构前后功能、数据、接口、测试完全一致

*   多步骤任务先输出简短执行计划，分步落地、分步验证

## 二、Django 项目基础编码规范

### 1. 基础环境与语法

*   遵循 Python PEP8 规范，4空格缩进，代码编码统一 UTF-8

*   基于项目当前 Django 版本开发，不擅自使用高版本废弃/新增特性

*   优先使用 **Django 原生API**，非必要不引入第三方库替代原生能力

*   核心业务代码必须添加类型注解，提升可维护性

### 2. 统一命名规范

*   项目、App、文件、函数、变量：**snake\_case**

*   Model、Serializer、Form、ViewSet 类：**PascalCase**

*   常量、配置参数：**UPPER\_SNAKE\_CASE**

*   数据库字段：snake\_case，与模型属性严格对应

### 3. Django 核心开发约束

*   数据库查询优先使用 Django ORM 原生语法，禁止裸写大量SQL语句

*   禁止忽略异常，必须捕获具体异常类型，不使用裸 except:

*   查询数据优先使用 get、filter、exists，避免重复查询、全表查询

*   批量数据操作使用 bulk\_create、bulk\_update，减少数据库IO

*   敏感数据（密码、手机号）必须加密处理，禁止明文存储、明文返回

*   接口参数校验优先使用 Serializer/Form，不手写冗余校验逻辑

## 三、Django 项目模块化分层规范（核心强制）

严格遵循 Django 官方 MVT 架构，所有代码必须归位，**禁止跨层乱写、职责混杂**，杜绝视图写业务、模型写逻辑的混乱问题。

### 1. 标准 Django 项目目录结构

```plain
project_root/
├── manage.py
├── project_name           # 项目根配置目录
│ ├── __init__.py          
│ ├── settings             # 拆分多环境配置
│ │ ├── base.py            # 公共基础配置
│ │ ├── dev.py             # 开发环境
│ │ └── prod.py            # 生产环境
│ ├── urls.py              # 全局主路由
│ ├── asgi.py              
│ └── wsgi.py              
├── apps/                  # 统一业务App存放目录
│   ├── common/            # 公共工具模块App
│   └── xxx_biz/           # 其他业务模块
├── docs                   # 文档
├── downloads              # 下载
├── logs                   # 运行日志
├── media                  # 用户上传资源
├── requirements.txt       
├── scripts                # 脚本工具
├── static                 # 静态资源
├── static_collect         # 静态资源目录,保存为Nginx收集的静态文件
├── templates              # 模板文件
├── tests                  # 全局测试用例
├── utils                  # 全局通用工具
└── CLAUDE.md

```

### 2. 全局目录职责定义

#### project\_name 项目配置目录

*   只负责项目全局配置、路由分发、服务启动配置

*   拆分 base/dev/prod 多环境配置，区分开发/生产参数

*   **禁止编写任何业务逻辑、数据处理逻辑**

#### apps 业务App目录（核心业务层）

所有业务模块统一收纳在 apps 下，一个业务对应一个独立App，每个App内部严格分层：

```plain
apps/demo/
├── __init__.py
├── admin.py      # 后台站点配置
├── apps.py       # App注册配置
├── models.py     # 数据库模型、数据字段定义
├── serializers.py# DRF序列化、参数校验
├── views.py      # 视图/接口逻辑、请求响应处理
├── urls.py       # 模块子路由
├── services.py   # 核心业务逻辑
├── forms.py      # 表单校验
└── tests.py      # 模块单元测试

```

#### 各App内部文件强制职责拆分

*   **models.py**：仅定义数据表结构、字段、元数据、基础钩子方法，**不写复杂业务逻辑**

*   **serializers.py**：仅负责参数校验、数据序列化、字段转换、嵌套校验，不处理业务流程

*   **views.py**：仅接收请求、权限校验、调用业务方法、返回响应，**禁止写复杂业务逻辑**

*   **services.py**：收敛当前模块所有核心业务逻辑、数据处理、流程编排，视图只调用不实现

*   **urls.py**：仅注册当前模块路由，不编写任何业务代码

*   **admin.py**：仅配置后台展示、编辑、过滤规则

#### utils 全局工具目录

*   存放全局通用工具：加密、时间格式化、参数解析、通用校验、异常封装

*   工具方法无业务侵入、可全局复用，禁止存放专属业务逻辑

#### tests 测试目录

*   按业务模块划分测试文件，覆盖接口、业务逻辑、边界场景

*   新增功能必须配套新增测试用例

## 四、Django 专属强制规则

### 1. 路由规范

*   全局主路由只做分发，具体业务路由下沉到各App内部urls.py

*   DRF接口优先使用 ViewSet + Router 自动生成路由，减少手写路由

*   路由命名清晰，遵循 业务模块-功能 命名规则

### 2. 模型规范

*   所有Model必须定义明确字段类型、长度、默认值、null/blank属性，禁止模糊定义

*   公共字段（创建时间、更新时间、创建人）抽取为抽象基类模型

*   数据库变更必须生成迁移文件，禁止手动改库、跳过迁移

*   字段注释清晰，标注字段用途、业务含义

### 3. 接口与响应规范（DRF）

*   所有接口统一响应格式，包含 code、message、data、status

*   参数校验全部交由 Serializer 完成，不重复校验

*   权限控制优先使用 DRF 权限类，不手写权限判断

*   分页、排序、过滤使用 DRF 原生组件，不手写逻辑

### 4. 业务分层红线

*   **绝对禁止**：views 中写大量业务计算、数据处理、嵌套判断逻辑

*   **绝对禁止**：models 中写接口响应、权限判断、复杂流程逻辑

*   **绝对禁止**：utils 工具混入专属业务逻辑

*   所有复杂业务必须下沉至对应模块 services.py，保证视图轻薄

### 5. 规范生效标准
*   所有代码改动满足：贴合Django MVT架构、分层清晰、职责单一、改动最小、无过度设计、无无效改动。
*   编码前主动澄清疑问，编码后功能稳定、数据正常、接口规范、可测试可维护，完全贴合项目现有技术架构与代码风格。

## 五、工程约束与强制禁止事项（AI 严格遵守）

*   禁止引入 Vue / React / ElementUI / AdminLTE 等其它前端框架

*   禁止改造为前后端分离架构

*   禁止后期使用 AbstractUser 改造用户体系

*   禁止使用 pip 管理依赖，所有依赖操作必须使用 uv

*   禁止修改现有多环境配置架构

*   禁止替换当前 mysql\.connector 兼容驱动方案

*   禁止擅自修改 settings 全局核心配置、数据库配置、中间件、权限配置

*   禁止擅自新增、删除、拆分App目录，新增业务必须遵循现有分层结构

*   禁止随意新增第三方依赖，如需引入，先说明用途与必要性

*   禁止提交明文密钥、密码，禁止硬编码敏感配置，敏感数据统一从环境变量读取

*   禁止编写冗余调试代码、print日志、无效注释

*   禁止过度封装、重复造轮子，优先复用Django/DRF原生能力

## 九、项目架构规范（严格遵守）

### 1. 配置架构

- 采用 **多环境分层配置**：base.py + dev.py + prod.py

- 所有敏感配置（SECRET_KEY、数据库账号密码）统一从 `.env` 读取，**禁止硬编码**、禁止默认兜底值

- 开发环境与生产环境静态文件、调试模式、存储策略完全差异化配置

### 2. 数据库 &amp; 用户体系规范

- 项目初始化后已存在原生 auth_user 表与存量数据

- **禁止后期使用 AbstractUser 替换原生用户表**（会迁移报错、数据丢失）

- 用户字段扩展统一采用 **OneToOne 一对一 Profile 扩展方案**

- 数据库字符集统一：utf8mb4，排序规则：utf8mb4_unicode_ci

### 3. 静态文件规范

- **STATIC_URL** 全局统一为 `static/`

- 开发环境通过 STATICFILES_DIRS 读取项目根目录 static 公共资源

- 生产环境使用 STATIC_ROOT + collectstatic 归集静态资源

- 静态文件存储引擎区分环境配置，生产开启哈希缓存策略

### 4. 项目开发规范

- 功能模块按 APP 拆分，单一职责原则，一个业务模块对应一个 APP

- 所有页面使用 Django 模板体系，继承公共布局模板

- 数据交互优先使用 Ajax 局部刷新，禁止整页刷新

- 手写原生 MySQL 代码与 Django ORM 可共存，互不冲突
